# CLAUDE.md generation — phase reference

You are generating a best-practice CLAUDE.md for the user's project.
Read all three docs: requirements.md, design.md, and tasks.md.

If a CLAUDE.md already exists, APPEND a clearly marked section.
Do NOT replace or delete existing content.

## What to include

A good CLAUDE.md gives Claude Code everything it needs to assist
effectively from the first session. Keep it under 200 lines.

### Project overview (from requirements)
- One-paragraph description: what this project does and who it's for
- Link to specs: "See @docs/requirements.md for full spec"

### Tech stack (from design)
- Language, runtime, framework
- Key dependencies
- Any external APIs or services

### Commands (from design + tasks)
- How to install dependencies
- How to build / compile
- How to run in development
- How to run tests

### Project structure (from design)
- The directory layout
- What goes where
- Which files are entry points

### Code style rules (from design + user preferences)
- Language-specific conventions
- Import style
- Error handling approach
- Testing conventions

### What NOT to do
- Boundaries from the out-of-scope section of requirements
- Any constraints from the design (don't modify X, always use Y)

## What NOT to include

- Detailed phase-by-phase instructions (those belong in skills)
- Copy-pasted content from the spec docs (link to them instead)
- Generic advice that any LLM already knows
- More than 200 lines (move detail to .claude/rules/ files)

## Confidence notes

If any section of the CLAUDE.md is based on low-confidence design
decisions, add a comment:
```
<!-- Confidence: medium — architecture pattern needs validation -->
```

## Output

Write using the template in assets/claude-md-template.md.
Save to the project root as CLAUDE.md.

If one already exists, add a section header:
```
## Project context (generated by repo-onboard)
```
And append below the existing content.
